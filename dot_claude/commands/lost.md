ユーザーはいま認知負荷が高く、セッションの流れについていけなくなっている。このコマンドの目的は **「何をアーティファクトにするか」の方向決めをユーザーではなく Claude が引き受ける** こと。

## 大原則

- **チャットに長文を書かない。** 混乱している人に追加のテキストを浴びせるのは本末転倒。説明はアーティファクト側に置き、チャットは最小限にする。
- **recall ではなく recognition。** 「何がわからないですか?」と自由記述で聞き返すのは禁止。Claude 側が仮説を立て、ユーザーは選ぶだけにする。
- **新しい用語を作らない。** アーティファクト内では、この会話で実際に使った言い回し・ファイル名・コマンド名をそのまま使う。言い換えると照合コストが増える。

## Step 1: 混乱ポイントの特定

`$ARGUMENTS` が指定されていればそれを混乱ポイントとして扱い、Step 2 へ進む。

未指定の場合は、直近の会話（特に自分の長い説明・提示した選択肢・方針転換・エラーからのリカバリ）を読み返し、**ユーザーが迷子になっていそうなポイントの仮説を 2〜4 個** 立てて AskUserQuestion で提示する（multiSelect: true、複数該当は普通にある）。

選択肢の書き方:
- 一般論ではなく **この会話の具体的な内容** に言及する。悪い例:「技術的な説明が難しかった」/ 良い例:「webhook の署名検証まわりの説明（HMAC のあたり）から追えていない」
- 各選択肢の description に、それが会話のどの時点の話かを一言添える
- ユーザーは「Other」で自由記述もできるので、選択肢で網羅しようとしなくてよい

## Step 2: 混乱タイプ → アーティファクト形式の対応表

選ばれたポイントを以下のタイプに当てはめ、形式を決める。複数選ばれた場合は 1 つのアーティファクトにセクション分けで収める（アーティファクトを複数作らない — タブを増やすのも認知負荷）。

| 混乱のタイプ | 出すアーティファクト |
|---|---|
| 全体のどこにいるか迷子 | 現在地ダッシュボード: ゴール / 完了したこと / いま作業中のこと / 次にやること / **ユーザーの判断待ちの未決事項** |
| 説明された概念・構造が掴めない | 図解（構成図・フロー図・関係図）。文章での再説明ではなく空間配置で示す |
| 選択を迫られたが判断材料が足りない | 選択肢の比較表 + それぞれのトレードオフ + 推奨とその理由 |
| 何がどう変更されたか追えない | 変更サマリ: ファイル別に「何を」「なぜ」。before/after が効く場合は並置 |
| 専門用語・登場人物が多すぎる | グロッサリー: 用語 / 一言定義 / **この会話の中で何の役割か** |

どのタイプでも共通:
- 選ばれた混乱に **関係ない情報は載せない**。網羅性より削ぎ落とし。全部盛りのダッシュボードは混乱を増やすだけ
- 未決事項（ユーザーの判断を待っているもの）があれば、タイプを問わず必ず目立つ位置に置く。迷子の最大の原因は「自分が何を求められているか」の見失いであることが多い

## Step 3: アーティファクト生成

1. **必ず先に `artifact-design` スキルを読み込む**（Artifact ツールの必須手順）
2. HTML はスクラッチパッドディレクトリに書く。日本語で書く
3. favicon は 🧭 で固定。タイトルは「〇〇の現在地」「〇〇の整理」のような短い日本語
4. 同一セッションで 2 回目以降の /lost は、新規作成ではなく **前回のファイルを更新して同じ URL に再デプロイ** する（ユーザーのタブを増やさない）

## Step 4: 完了報告

チャットには 1〜2 文だけ書く: 何を図解したか + ユーザーに判断してほしいことが残っているならそれ 1 点。アーティファクトの内容をチャットで再説明しない。
